One of the last projects I worked on at Criteo was building and running a custom Kubernetes scheduler plugin. In this post, I want to share what I learned and how you can do the same.

Kubernetes Scheduler Architecture Overview¶

Understanding the Kubernetes scheduler architecture is essential for customizing it’s behavior. In simple terms, the scheduler picks the best node for each Pod based on available resources and the rules you set in the Pod spec (requests, limits, affinity, taints, and more).

In more complex terms, picking the best node part consists of filtering, scoring nodes and picking the node with best score.

Kube Scheduler Configuration Example¶

Kubernetes allows customization of the default scheduler through configuration profiles. These profiles can define scheduling policies tailored to your specific needs:

apiVersion: scheduler.k8s.io/v1
kind: Scheduler
metadata:
  name: my-custom-scheduler
spec:
  profiles:
    - name: profile1
      plugins:
        score:
          enabled:
            - name: PluginName
        filter:
          enabled:
            - name: AnotherPlugin

In this example, you can see the use of specific scoring and filtering plugins, which help in determining the best node based on your criteria.

Running Multiple Schedulers¶

For debugging purposes, it’s possible to run several schedulers concurrently in your environment. While this is not recommended for production due to complexity, it’s a great way to test different scheduling algorithms and configurations to find the optimal setup. Keep in mind that this should be approached with caution and mainly for development.

Kubernetes-Sigs/Scheduler-Plugins Repository¶

The Kubernetes community has developed a wealth of plugins, both default and community-contributed. You can explore the kubernetes-sigs/scheduler-plugins repository, which provides a variety of scheduling solutions beyond the default.

Existing Plugins Shortlist¶

Before investing time in developing your own plugin, it might be beneficial to check the existing plugins. The community has created a diverse range of plugins that could potentially meet your needs. Always refer to the complete list of available plugins to see what’s already available!

Plugin Types¶

Kubernetes scheduling plugins come in several types, including but not limited to:

  • Score Plugins: Determine how well a node fits a pod.
  • Sort Plugins: Order nodes based on their suitability for a pod.
  • Enqueue Plugins: Handle the queuing of pods for scheduling.

Understanding these plugin types can help you better navigate the scheduling process and enhance your Kubernetes experience.

Testing Your Scheduler¶

It’s equally important to rigorously test your scheduling logic. Below are a couple of strategies you can employ:

  • Scheduler Simulator: This tool allows you to simulate scheduling decisions and see how your algorithms perform in various scenarios.
  • Testing in a Kind Cluster: Deploy your Kubernetes cluster locally using Kind (Kubernetes in Docker). This environment provides a convenient setup for testing your custom scheduler under realistic conditions.

Implementing a Custom Scheduling Algorithm¶

If you decide you need a custom scheduling algorithm for Kubernetes, start by leveraging the plugins architecture. Here’s a high-level approach to implementing it:

  1. Define Your Algorithm: Clearly outline how you want to score or filter nodes based on your specific use case.
  2. Create a Plugin: Build your plugin matching the k8s scheduler’s interface for your defined algorithm.
  3. Test Thoroughly: Use the aforementioned testing tools to ensure your new scheduling logic works under different conditions.

By following these steps, you can integrate your custom logic effectively within Kubernetes.
